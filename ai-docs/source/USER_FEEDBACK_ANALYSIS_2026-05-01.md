# User feedback analysis — 2026-05-01

Источник: `USER_FEEDBACK_2026-05-01.md`. Анализ корневых причин и
маппинг в задачи Wave 11.

---

## Сводка

Третий прод-тест после Wave 10 показал блокирующие баги в основном цикле
«создать партию → подтвердить участие → начать → финализировать».
Часть проблем — security (восстановление пароля по нику + видимая
подсказка секретного слова).

Wave 11 — строго hotfix-волна, ничего нового.

---

## Пункты фидбэка

### 1. «Тех кого изначально пригласили в партию при создании, попадают в неё как под вопросом»
**Источник в фидбэке:**
> «тех кого изначально пригласили в партию при создании, попадают в нее
> при создании как под вопросом, а то так сейчас собираешь пати, добавляешь,
> а создается пустая.»

**Корневая причина:** `CreateSessionPage` для обычной партии
(`entryMode === 'planned'`) после `createSession()` крутит цикл
`addParticipant(...)`, который создаёт `Participation` записи **до старта
партии**. ADR-0017 говорит, что до старта ростер ведётся через `SessionInvite`,
а `Participation` создаётся только в `start_session()`. Конфликт между двумя
flow:

1. На странице «Начать» юзер видит инвайты, формирует `factions_assignment`,
   нажимает «Начать».
2. `start_session()` пытается `bulk_create` `Participation` → UNIQUE constraint
   на `(session_id, faction_id)` → 400.

Симптом «создается пустая» = клиентский цикл `addParticipant` сейчас не
успевает / падает молча, и в карточке ничего не видно.

**Решение:**
- Фронт перестаёт вызывать `addParticipant` при `entryMode='planned'`.
- Вместо этого вызывает `inviteUser(sessionId, { user_id, desired_faction,
  rsvp_status: 'maybe' })`.
- Default `maybe` потому что: «создатель собрал состав, но участники
  должны подтвердить». Это нейтральный дефолт, не «иду» (пользователь сам не
  тыкал) и не «invited» (визуально пустой статус).

**Задача:** F-230. Архитектурное решение: ADR-0019.

### 2. «Кнопка под вопросом не нажимается пока не нажмёшь "я иду"»
**Источник в фидбэке:**
> «Кнопка под вопросом не нажимается пока не нажмешь я иду, только потом как
> будто разблокируется, нужно чтобы сразу можно было и ты вступаешь под
> вопросом.»

**Корневая причина:** В `RsvpBlock.handleRsvp`:
```ts
if (!myInvite) {
  if (!user || status !== 'going') {
    return  // <— это блокирует maybe и declined
  }
  ...
  await selfInviteMutation.mutateAsync()  // <— берёт default 'going' с бэка
}
```
Поэтому без существующего инвайта работает только `going`. Чтобы попасть в
`maybe`, надо сначала создать инвайт через `going`, затем переключить.

**Решение:**
- `selfInvite()` API расширяется: `{ rsvp_status?: RsvpStatus,
  desired_faction?: string | null }`.
- Backend `self_invite` уже поддерживает `rsvp_status` через ADR-0019.
- В `RsvpBlock.handleRsvp` убирается ограничение «только going без инвайта».
  Если инвайта нет — вызывается `selfInvite(sessionId, { rsvp_status })`.

**Задача:** F-231.

### 3. «В меню редактирования партии, если пригласить кого-то — он должен отобразиться как под вопросом»
**Источник в фидбэке:**
> «И все еще есть путаница, в меню редактирования партии есть меню пригласить
> игроков, туда так же если пригласить кого то он должен отобразится в партии
> как под вопросом.»

**Корневая причина:** Та же что и пункт 1 — `EditSessionPage` тоже зовёт
`addParticipant` для новых записей в форме.

**Решение:** В `EditSessionPage.onSubmit`:
- Удалять `removeParticipant`/`updateParticipant` для существующих participations
  (этих participations не должно быть на planned сессии после Wave 11).
- Для новых seed'ов в форме — вызывать `inviteUser` с `rsvp_status: 'maybe'`.

**Задача:** F-230 (объединена).

### 4. «Начать партию по обычному не получается» — POST /sessions/10/start/ → 400
**Корневая причина:** Та же что в пункте 1 — orphan `Participation` от старого
flow создания/редактирования. `start_session()` в сервисе падает на UNIQUE
constraint.

**Решение:**
- F-230 убирает источник проблемы (новые сессии).
- T-173 management команда `cleanup_orphan_participations` чистит уже
  созданные кривые сессии на проде.

### 5. «GET /sessions/10/start/ → 500 (MethodNotAllowed как unhandled)»
**Источник в фидбэке:**
```
GET /api/v1/sessions/10/start/
HTTP 500 Internal Server Error
{
  "exception_class": "MethodNotAllowed",
  "exception_message": "Метод \"GET\" не разрешен."
}
```

**Корневая причина:** В `ErrorHandlingMixin` (apps/games/views.py,
apps/ratings/views.py) `MethodNotAllowed` не обрабатывался отдельно — он
падал в catch-all 500 handler. То есть API возвращал статус **500** на запрос,
который должен быть **405**.

**Решение:** Добавить явную ветку для `MethodNotAllowed` в обоих
`ErrorHandlingMixin`:
```python
if isinstance(exc, MethodNotAllowed):
    return build_error_response(
        code="method_not_allowed",
        status_code=status.HTTP_405_METHOD_NOT_ALLOWED,
        ...
    )
```

**Задача:** T-171.

### 6. «При вводе пароля — кнопочку чтобы пароль показывался»
**Источник в фидбэке:**
> «При вводе пароля (логин, регистрация, восстановление пароля) чтобы пароль
> при нажатии на кнопочку показывался.»

**Корневая причина:** Стандартный UX-нюанс. Поле `<input type="password">`
без toggle-eye — людям тяжело проверить, что они ввели правильно.

**Решение:** Создать `PasswordInput` компонент (вокруг `Input` из
`components/ui/`) с иконкой `Eye/EyeOff` справа, click → toggle между
`password` и `text`. Использовать на:
- `LoginPage` — поле пароля.
- `RegisterPage` — `password` + `password_repeat`.
- `PasswordResetPage` — `new_password` + `new_password_repeat`.
- `MyProfilePage` (секция изменения пароля) — `current_password` +
  `new_password` + `new_password_repeat`.

**Задача:** F-232.

### 7. «Восстановление пароля только по почте + убрать плейсхолдер секретного слова»
**Источник в фидбэке:**
> «Очень важно, при восстановлении пароля у незалогиненого пользователя
> убрать плейсхолджер секретного слова, это бред его там показывать +
> восстановление только по почте, но не по нику, по нику легко взломать,
> секретное слово то все знают.»

**Корневая причина (security):**
1. В `PasswordResetPage` placeholder секретного слова `"lovecraft"` (или
   аналогичный) фактически раскрывает значение слова — кто угодно может
   зайти на `/password-reset` и прочитать.
2. Recovery по нику: ник публичный, секретное слово знают все в группе →
   любой участник может сбросить пароль другому.

**Решение (двуслойное):**
- **Backend (T-172):** `reset_password()` принимает только `email`. Поле
  переименовано из `login` в `email`, тип `EmailField`. Recovery по нику
  технически невозможен.
- **Frontend (F-233):** `PasswordResetPage`:
  - Label «Email» вместо «Почта или ник».
  - `placeholder="you@example.com"`, `type="email"`.
  - В payload — `email` вместо `login`.
  - Поле «Секретное слово»: placeholder пустой, helper text нейтральный
    («Секретное слово сообщит администратор» или вообще убрать).
  - Тип компонента — `<input type="text">` (не password — оно действительно
    не секрет в смысле пароля, но и подсказывать его не нужно).

**Задачи:** T-172, F-233.

### 8. «POST /sessions/8/finalize-played/ → 400» (быстрая отметка)
**Источник в фидбэке:**
```
не могу создать партию ( быстрая отметка )
пишет — Ошибка валидации.
POST .../sessions/8/finalize-played/ 400 (Bad Request)
```

**Корневая причина:** `finalize_played_session()` в сервисе требует, чтобы в
сессии **не было `Participation`** (защита от двойной финализации). Сессия №8
была создана старым flow — `Participation` уже лежали → 400.

**Решение:**
- F-230 устраняет источник проблемы (новые сессии).
- T-173 чистит существующие.
- Сообщение об ошибке («В этой партии уже есть участники») остаётся как есть
  — оно корректное, просто после T-173 такого состояния не будет.

## Архитектурное решение

**ADR-0019:** До `start_session()` единственный способ попасть в ростер —
`SessionInvite`. `Participation` создаётся только в `start_session()` и
далее. Все frontend-формы планирования создают invites с `rsvp_status: 'maybe'`
по умолчанию.

Отдельное решение по recovery: только по email, ник запрещён.
