# User Feedback Analysis — 2026-04-27

**Source:** `ai-docs/source/USER_FEEDBACK_2026-04-27.md` (исходный текст владельца).
**Author:** architect.

Это **разбор** user-фидбэка после первого реального теста MVP. Заявок много, и они смешаны: критичные баги, новые большие фичи, мелкая полировка, доменные правила игры, требования к продукту.

Цель этого документа — **разобрать запрос на категории**, для каждой категории определить:
- что именно меняется,
- какой это уровень (баг / change request / новая фича / ADR-уровень),
- какие задачи/документы порождает,
- зависимости.

После этого — переписываем `BACKLOG`, `DATA_MODEL`, ADR-list и выдаём агентам.

## 1. Доменная модель — фундаментальный pivot

Юзер уточнил **правила игры**, и из них следует, что текущая модель **не соответствует**. Это критично — все остальные изменения зависят от пересмотра доменной модели.

### 1.1. Lifecycle сессии — `planned → in_progress → completed`

**Сейчас:** `planned → completed | cancelled`. Чтобы финализировать, должна быть `planned`. Юзер прямо жалуется: *«пишет что финализировать можно только planned, странно»*.

**Должно быть:** жизненный цикл сессии включает **активную игру** между планированием и финалом:

```
planned ─(start)─▶ in_progress ─(finalize)─▶ completed
   │                    │
   └─(cancel)──────────┴────────────▶ cancelled
```

**Семантика:**
- `planned` — план, RSVP, выбор желаемой роли. Состав может меняться.
- `in_progress` — игра идёт, фракции уже распределены, начали отмечаться раунды и события timeline. Замены игроков возможны (см. §3.4). Финализировать можно только из `in_progress`.
- `completed` — игра окончена, есть Outcome, голоса, статистика. Финализация запускается **только если по последнему раунду есть игрок с 7 замками**, а при равенстве замков — победа определяется треком трона.
- `cancelled` — отменена, в статистику не идёт.

**Породит:**
- `ADR-0009: Game session extended lifecycle`.
- Новый сервис `start_session`.
- Изменение `finalize_session` — принимает `in_progress`.
- Frontend wizard для перехода `planned → in_progress` (распределение ролей).

### 1.2. Rounds & trackers — большой новый домен

**Сейчас:** `Outcome.rounds_played` — одно число. Никакого реального трекинга.

**Должно быть:** в течение `in_progress` фиксируются **снимки состояния по раундам**. По факту это **новая полноценная модель**:

```
GameSession
└── RoundSnapshot (1..N)
    ├── round_number              (1..10, либо до 7 для quests/alternative)
    ├── happened_at
    ├── influence_throne          # трек власти 1: Железный трон
    ├── influence_sword           # трек власти 2: Вотчина
    ├── influence_court           # трек власти 3: Королевский двор
    │   └── каждый — упорядоченный список participation_id (1-е, 2-е…), без дублей
    ├── supply                    # supply.{participation_id} = 0..6
    ├── castles                   # castles.{participation_id} = 0..7
    └── wildlings_threat          # 0,2,4,6,8,10,12 (7 позиций)
```

Доменные правила:
- Места на каждом из 3 треков влияния — упорядоченный список participations. На одном месте только один игрок.
- `castles` — 0..7. 0 — красная зона, 7 — победа.
- `supply` — 0..6, может быть несколько игроков на одной отметке (поэтому это карта `participation_id → 0..6`, не упорядоченный список).
- `wildlings_threat` — 0/2/4/6/8/10/12 (7 шагов), стартовая позиция 4.
- Звёзды на королевском дворе вычисляются из числа игроков (см. §1.3).

**Породит:**
- `ADR-0010: Round snapshot model`.
- Backend: `apps/games/models.py::RoundSnapshot` + endpoints.
- Frontend: UI снапшота, таймлайн матча, drill-in в раунд.

### 1.3. Star slots на «Королевском дворе»

Из feedback: *«у определённых мест смотря сколько игроков играет есть определённое кол-во звезд (для 4+ игроков у первого места 3 звезды, у второго 3, у третьего 2, у четвертого 1 звезда. А в 4- игроков, 1 место три, 2 место два и третье место одно очко, остальные без звезд)»*.

**Это правило отображения, не хранения.** Звёзды вычисляются по позиции на треке `influence_court` и количеству игроков:

| Players | 1-е место | 2-е | 3-е | 4-е | 5-е | 6-е | 7-е | 8-е |
|---------|-----------|-----|-----|-----|-----|-----|-----|-----|
| 3       | 3         | 2   | 1   | —   | —   | —   | —   | —   |
| 4–8     | 3         | 3   | 2   | 1   | 0   | 0   | 0   | 0   |

Закладываем как утилиту во `frontend/src/lib/court-stars.ts`. Данные не дублируются.

### 1.4. Event decks (3 или 4)

Из feedback: *«если играем 6- игроков то 3 колоды, а если 7+ (режим с драконами только там может быть больше 6 игроков) то 4 колоды»*.

**В текущей модели `Deck`** хранятся колоды домов («Original», «Expansion A», ...). Это путаница: юзер говорит про **колоды событий** (event decks) — 1, 2, 3 (и 4-я в Mother of Dragons). Каждая колода = 10 фиксированных карт; при розыгрыше игрок выбирает карту из списка. Для квестов — 1-я колода другая, в Mother of Dragons добавляется 4-я.

**Reference Deck (колода домов) и Event Deck (колода событий) — разные сущности.** Текущий `reference.Deck` нужно либо переименовать в `HouseDeck`, либо вообще убрать (см. §1.5 — юзер хочет 2 варианта).

**Породит:**
- `ADR-0011: Event decks redesign`.
- Hardcoded event cards (по 10 на колоду, разные по режимам).
- Endpoint для drag/play event card в раунде.
- Frontend: dropdown «выбрать карту» в раунд-интерфейсе (или оставить пустым).

### 1.5. House Decks — 2 варианта

*«в Game sessions в поле Deck сделать только 2 варианта (Оригинальная колода и Альтернативная колода)»*.

Сейчас `Deck` имеет 3 записи: `original`, `expansion_a`, `expansion_b`. Должно стать **2: `original`, `alternative`**. Поле `Deck` относится к **колодам домов** (карты приказов и звёзд для каждой фракции).

**Простой fix в seed-миграции** — но breaking change, потому что есть сессии, ссылающиеся на `expansion_a/b`. У нас по факту нет prod-данных, поэтому миграция чистая.

### 1.6. Game modes — сжатые правила

Юзер свёл правила:

| Режим                 | Игроков     | Доступные дома                                       | Event decks |
|-----------------------|-------------|-------------------------------------------------------|-------------|
| Classic               | 3–6         | 3:`stark/lannister/baratheon`; +1`greyjoy`; +1`tyrell`; +1`martell` | 3 (все стандартные) |
| Feast for Crows       | 4 (ровно)   | `arryn`, `stark`, `lannister`, `baratheon`            | 3 (1-я своя) |
| Dance with Dragons    | 6 (ровно)   | classic-составь без `arryn` и `targaryen`             | 3 (стандарт) |
| Mother of Dragons     | 4–8         | все 8; **`targaryen` обязателен**                     | 4 (добавляется) |

**Текущее состояние некорректно по нескольким пунктам:**
- `alternative` сейчас «Танец с драконами 4–4», должно быть `dance_with_dragons` 6–6.
- `quests` (Feast for Crows) — состав фракций не валидируется.
- `dragons` (Mother of Dragons) — нет ограничения «обязательно targaryen».

**Породит:**
- `ADR-0012: Game mode rules engine`.
- Сервис валидации `validate_session_setup(mode, participants)` — единый источник правил.
- В seed: 4 режима с правильными границами, `expected_factions: list[str]` или `forbidden_factions: list[str]` или `required_factions: list[str]`.

### 1.7. Фракции — 8, не 9; favourite — косметика

Юзер ясно: *«фракция выбранная в личном кабинете это просто твои фавориты, в дальнейшем просто темку сделаю для каждой фракции свои, то есть это только картинка ни никакой другой функционал, особенно при выборе и входе в игру не играет. Поэтому всю механику завязанную на фракциях нужно еще раз пересмотреть всю.»*

Сейчас всё корректно: `Profile.favorite_faction` — отдельно от `Participation.faction`. Но нужно **зафиксировать в DATA_MODEL** что это два разных значения и убедиться что фронт это нигде не путает (например, не предзаполняет `Participation.faction = profile.favorite_faction`).

8 фракций уже подтверждено в CR-002. Targaryen — есть. Tully — нет.

## 2. Auth & registration — много правок

### 2.1. Registration: secret word + auto-active

Из feedback: *«При регистрации сделать бы повтор пароля и возможность ввести секретное слово (lovecraft) регистр не важен, то не нужно подтверждение от админа, сразу активный»*.

**Должно быть:** при регистрации вводится `password`, `password_repeat`, опциональное `secret_word`. Если `secret_word.lower() == "lovecraft"` (хранится в env) — `is_active=True` сразу. Иначе — `is_active=False` + ждёт approve.

### 2.2. Password reset

Из feedback: *«страницу восстановления пароля, где нужно ввести почту или ник + секретное слово (lovecraft) регистр не важен»*.

**Должно быть:** `POST /api/v1/auth/password/reset/` с `{login, secret_word, new_password, new_password_repeat}`. Login — это **либо email, либо ник** (см. §2.4). Если `secret_word` совпадает — пароль сбрасывается, иначе 400.

### 2.3. Change password (logged-in)

Из feedback: *«Кнопка поменять пароль со всей логикой по смене дальше»*.

**Должно быть:** `POST /api/v1/auth/password/change/` с `{current_password, new_password, new_password_repeat}`. Стандартный change-password flow в `MyProfilePage`.

### 2.4. Login by email **или** nickname

Из feedback: *«Можно входить как по нику + пароль, так и по емейл + пароль»*.

**Должно быть:** в `LoginView` поле называется `login` (не `email`). Сервис ищет пользователя по `User.email__iexact` ИЛИ `Profile.nickname__iexact`. Если нашёл — стандартный `authenticate`.

### 2.5. Avatars не отображаются — bug

Корень проблемы: `serializer.get_current_avatar` возвращает `avatar.generated_image.url` — это **относительный путь** `/media/avatars/...`. Фронт делает `<img src="/media/avatars/...">` и идёт на `localhost:5173/media/...` — Vite ничего не знает, 404.

**Fix:** `request.build_absolute_uri(avatar.generated_image.url)` → возвращается `http://localhost:8000/media/avatars/...`. Vite проксирует `/media` (или фронт получает абсолютный URL — оба варианта работают).

Альтернатива: настроить Vite proxy на `/media` тоже. Это даже проще.

## 3. Match flow — большой новый функционал

### 3.1. Invitations & RSVP

*«изначально если событие запланировано, можно в него вступить или отказаться, чтобы знали что ты не придешь или ничего не жать, тебя туда могут пригласить, тебе придет уведомление с подтверждением внутри сайта»*.

**Должно быть:**
- Создатель сессии может **пригласить** конкретных игроков (по нику).
- Любой авторизованный игрок может **сам вступить** (RSVP=going) если сессия публична.
- Возможные статусы: `going / maybe / declined`. Без статуса = «не отметился».
- Приглашение → in-app notification.

Это **изменение `Participation`** — раньше `Participation` создавался только при добавлении в состав. Теперь нужен пред-состав («намерение»):

**Решение:**
- `Participation` остаётся «реальный участник» с фракцией.
- Добавляется новая модель `SessionInvite(session, user, rsvp_status, desired_faction, invited_by, created_at)`.
- При переходе `planned → in_progress` админ преобразует invites c `going` в `Participations`, распределив фракции.

### 3.2. Желаемая роль (desired faction)

*«ты можешь указать желаемую роль, она где-то рядом с ником отобразится, это ни на что не влияет, просто показать за кого хочешь играть, у нескольких может быть одинаковая»*.

`SessionInvite.desired_faction` — необязательное, **не уникальное** в сессии. Используется только как UI-подсказка, при распределении фракций админом.

### 3.3. Random faction assignment

*«нажать кнопку рандом и места рандомно распределятся, потом в дальнейшем добавим умное распределение в зависимости от статистики»*.

Кнопка «Рандомное распределение» — backend service `randomize_factions(session)`, использует доступные для режима фракции, учитывает `targaryen_required` для Mother of Dragons. Гарантирует уникальность (CHK уже есть в models).

### 3.4. Replacement (замена игрока в середине)

*«Посреди игры игрок может выйти и другой на его место зайти (замена) и как-то тоже в истории это отмечать»*.

Когда `status=in_progress`:
- Сервис `replace_participant(session, old_user, new_user)`.
- В `Participation` сохраняется FK на user, но добавляется поле `replaced_at` и optional `replaced_by_participation` self-FK.
- В timeline появляется событие `participant_replaced`.

**Альтернатива:** не хранить self-FK, а просто новые `Participation` с `joined_at_round` и старые `left_at_round`, считая что фракция «передаётся». Простоту дает первый вариант.

### 3.5. Cancel — кнопка из UI

*«партия можно отменить если никто не собрался»*. Уже есть сервис `cancel_session`, но **на фронте нет кнопки** в `MatchDetailPage`. Просто wire button.

### 3.6. Finalize — pre-check на 7 замков

*«если игра окончена можно нажать кнопку закончить игру, только инфа по победителю идет из последнего результата раунда, то есть там должен быть победитель с 7 замками, иначе разворот и нельзя закрыть игру пока не отметишь правильно раунд»*.

`finalize_session` сейчас принимает `places` напрямую. Должен:
1. Брать **последний RoundSnapshot**.
2. Проверять что у кого-то `castles == 7`.
3. Если несколько игроков с 7 замками — победитель тот, у кого выше место на `influence_throne`.
4. Раздать `place=1..N` по приоритету: 7 замков → throne → other castles → throne tiebreaker.

**Породит изменение `finalize_session` API:** payload меняется — agent просто **подтверждает** результат, а не вводит места вручную.

### 3.7. Wildlings raid action

*«создатель партии или админ может разыграть действие нападение одичалых, тут каждый сам за себя или создатель/админ может заполнить какую ставку сделал каждый игрок, затем выбрать победу или поражение и выбрать из определённого списка [...] и в истории этого матча в хронологии появится такой момент»*.

Это **timeline event**:
- Pre-action: список ставок по игрокам.
- Outcome: `win | loss`.
- Outcome card slug: из enum (заглушка пока, потом юзер пришлёт список).
- Wildlings increase/decrease на треке после исхода.

### 3.8. Clash of kings (битва королей)

*«битве королей (3 раунда ставок, кто ставит больше получает выше трек (трон, меч, звезды) как и нападение одичалых это активируемое действие, после которого можно внести данные в 3 этапа за каждый трек, кто сколько поставил и какое место»*.

3 этапа в timeline event: для каждого из 3 треков (throne, sword, court) — ставки игроков и итоговый порядок мест. По факту это **специальный round event** который пересоздаёт `influence_*` треки.

### 3.9. Match Timeline (хронология) + Chronicler в чате

*«В чате отображались еще важные игровые моменты (замены игроков, окончания раунда (при клике можно в них провалиться) они как-то визуально помечены что это летописец войны (система)»*.

Каждое значимое действие → событие в timeline + сообщение от системного «летописца» в чате. Можно отключить видимость летописца локально.

**Модель:**
- `MatchTimelineEvent(session, kind, payload, happened_at, round_snapshot=null)`.
- `kind ∈ {round_completed, wildlings_raid, clash_of_kings, event_card_played, participant_replaced, session_started, session_finalized}`.
- `payload: JSONField` — структура зависит от `kind`.
- Альтернатива: отдельные модели на каждое событие (полиморфизм). JSON проще на этом этапе, при этом тип строго типизируется на TS-стороне.

В чате (`MatchComment`) добавляется поле `kind ∈ {user, chronicler}`, чтобы UI знал, что это системное.

## 7. Решения, которые я (architect) принимаю самостоятельно

Чтобы не блокировать поток на consultation, фиксирую без обсуждения:

1. **Timeline events: JSON payload** (не отдельные модели на kind). Меньше boilerplate, на TS типизируем discriminated union.
2. **Round snapshots: один объект-снимок в конце раунда**, а не отдельные модели для каждого трекера. Снимок целостный, immutable, после создания не меняется.
3. **SessionInvite vs Participation:** оставляем разные модели. Participation = реальный участник; SessionInvite = намерение/RSVP.
4. **Replacement: pair Participation A→B** через `replaced_by_participation` self-FK. Старая запись остаётся в истории, у неё `left_after_round`, новая берёт фракцию.
5. **Login by email-or-nickname:** на бекенде `LoginSerializer.login` принимает строку, сервис сам определяет тип. Frontend label: «Email или ник».
6. **Secret word:** хранится в env `REGISTRATION_SECRET_WORD`. Сравнение case-insensitive. Если env не установлен — поведение «без секрета» (всегда требует approve).
7. **i18n: пока без библиотеки.** Все строки русские, явно. Если позже потребуется английский — введём `react-intl` отдельной задачей. Не тащим i18next с собой ради будущей фичи.
8. **Custom error pages: 404, 500, 403, network-error.** Пока 4 страницы. По дизайну — fantasy-стиль, та же палитра.
