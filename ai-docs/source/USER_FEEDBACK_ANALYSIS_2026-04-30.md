## 1. Концептуальный pivot: «участники» = «инвайты»

### Проблема, как описал юзер

> «Если ты нажимаешь присоединится ты добавляешься в участники, но не появляешься в турнирной таблице, почему? это разное? он должен и туда попадать. Нужно как то разрулить эту сложность, вот людей пригласили, они тут в итоговой таблице какой то отображаются, в участниках, а те кто вступил попадают в участники только и зачем эти два меню? Давай так, итоговой таблицы можно скрыть до того как начнется раунд. вот и тех кого пригласили типа добавляются в участники, им приходит уведомление и они тоже должны прожать иду или не иду, если не иду то удаляется из участников, так же чтобы было видно нажал он что то или нет, то есть показать если он не отреагировал»

> «Кнопка присоединится дублирует функционал я иду»

### Текущее состояние

В системе **две** таблицы:
- `Participation` — реальный участник партии (фракция, место, замки). Создаётся при `start_session`.
- `SessionInvite` — инвайт/RSVP (going / maybe / declined / invited). Существует во время `planned`.

Frontend показывает **обе таблицы рядом в `MatchDetailPage`** при `status=planned`:
- «Турнирная таблица» (Participations) — пустая до старта.
- «Участники» (SessionInvites) — заполняется по мере RSVP.

Это **сбивало юзера** — он не понимал зачем два списка одного.

### Решение

`SessionInvite` остаётся как сущность под капотом (это правильная модель — см. ADR-0013), **но UI показывает только её** во время `planned`:
- Один список «Участники» (= invites).
- Кнопки «Иду / Под вопросом / Не иду» возле своей строки.
- Иконка/подпись «не отреагировал» для `invited`-статуса.
- Pre-турнирная таблица (`Participation`) **не показывается** до `start_session`.
- При `in_progress` / `completed` появляется реальная таблица.

**Кнопка «Присоединиться»** — она делает self-invite (`POST /invites/me/` с `going`). По факту это **синоним** «Я иду» для незарегистрированных. Уберём её как отдельную кнопку — заменим на одну группу из 3 RSVP-кнопок, доступных любому залогиненному юзеру.

При нажатии «Не иду» — invite **либо переходит в `declined`** и **показывается серым** в списке («N не идут»), либо удаляется. Юзер хочет: «если не иду — удаляется из участников» — то есть скрывается из основного списка, но в свёрнутой подсекции «не пойдут» виден.

**Итог UX:**
```
┌─ Участники ───────────────────────────────┐
│ ✓ alice (Старк, going)        [Я иду* | …]│  ← мой row подсвечен, кнопки активные
│ ✓ bob (Ланнистер, going)                  │
│ ? carol (нет ответа)                      │  ← invited, серенькая
│                                           │
│ ⌄ Не пойдут (1)                            │  ← collapsible
│   ✗ dave (Баратеон, declined)             │
└───────────────────────────────────────────┘
```

→ ADR-0017 (мини): «Участники в UI = SessionInvite, не Participation».
→ T-140 (backend): API `GET /sessions/<id>/invites/` уже есть, но проверить что отдаётся **полная** инфа (включая `desired_faction`, и пользователь). Сериализатор может потребовать расширения.
→ F-210 (frontend): большой rework `MatchDetailPage` секция «состав партии».

## 2. RSVP не работает на ряде экранов

### Пункты юзера

> «кнопки я иду, под вопросом и не иду на экране партии дня тоже не работают»
> «Кнопка присоединится дублирует функционал я иду»
> «Чтобы в партии показывалось кто нажал я не приду и кто под вопросом»
> «И чтобы админ мог игроков выгнать, например неактивных приглашенных»

### Текущее состояние

В `HomePage.tsx`:
```ts
type RsvpState = 'going' | 'maybe' | 'not-going'
const [rsvp, setRsvp] = useState<RsvpState>('going')
```
Это **локальный state**, без вызова API. То есть кнопки нажимаются, цвет меняется, но на сервер ничего не уходит.

В `MatchDetailPage` — RSVP вызывает API через `RsvpBlock`. Должно работать.

### Решение

**F-211**: HomePage — если есть `next_match`, заменить self-rolled RSVP-кнопки на тот же `RsvpBlock` компонент что в `MatchDetailPage`. Один источник истины, один путь. Убираем дубль кода.

**F-210** (тот же что выше): отображение declined и invited (без ответа) с визуальной отметкой.

**T-141 (backend)**: дать creator/admin **удалять чужой инвайт без подтверждения**. Сейчас `withdraw_invite` есть, но `permission` его на стороне views надо проверить — возможно требует self-only. Расширить до `creator | admin`.

→ Wave 10 → F-211 (RSVP на главной), F-210 (RSVP rework + admin kick).

## 4. Head-to-Head — auto-pick

### Пункт юзера

> «В h2h тебе автоматом выбирало сначала тебя + кого то интересного по стате, но так же можно было заменить, тоесть по дефолту было как я описал»

### Текущее состояние

`HeadToHeadPage.tsx` читает `?user_a=&user_b=` из URL, без автоподбора.

### Решение

**F-222**: при первом открытии `/h2h` (без query params):
1. `user_a` = текущий залогиненный пользователь (если есть профиль).
2. `user_b` = «интересный» оппонент. Простая эвристика: из stats overview взять второго в leaderboard (или того, с кем у текущего юзера больше всего совместных игр / кого он не побеждал). Можно сделать как новый endpoint `GET /api/v1/stats/head-to-head/suggested/?for_user=<id>` который возвращает interesting_opponent_id.

→ T-152 (backend endpoint) + F-222 (frontend default).

## 6. Архитектурные решения, которые я (architect) принимаю

**ADR-0017**: «UI uses SessionInvite as the canonical 'roster' during planned phase».
- Participation продолжает существовать как backend сущность (создаётся при `start_session` через invites).
- UI в `MatchDetailPage` при `status=planned` показывает **только** invites.
- При `status in (in_progress, completed)` показывает **только** participations (с местами и замками).
- При `status=cancelled` — invites без RSVP-кнопок (read-only).

Это не ломает API контракт. Только UI меняет.

**Принимаю:** `original` остаётся дефолтной House Deck, но фронт запоминает в localStorage последний выбор пользователя и предлагает его дефолтом.

**Принимаю:** Default scheduled_at = `new Date()` (текущий момент в local timezone, который теперь Asia/Yekaterinburg).

**Принимаю:** Time zone — `Asia/Yekaterinburg` (GMT+5). Меняется в backend `TIME_ZONE`. Frontend форматтеры через `Intl.DateTimeFormat('ru', { timeZone: 'Asia/Yekaterinburg' })` — без `date-fns-tz`, чтобы не тащить новую зависимость.

**Не принимаю**: предложение «при declined — удалять из списка». Корректнее **скрывать в свёрнутую секцию «не пойдут (N)»**, чтобы creator/admin видел картину целиком и при необходимости мог пригласить кого-то ещё.
