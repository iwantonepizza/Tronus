# Report for T-015: accounts admin approval action

**Agent:** codex
**Date:** 2026-04-22
**Status:** `completed`

---

## Summary

Для `accounts.User` настроен кастомный `UserAdmin` с action `Approve selected users`, а сигналы дополнены логикой активации `False → True`: при апруве гарантируется наличие `Profile` и добавление пользователя в группу `player`. Повторный approve остаётся идемпотентным и не вызывает ошибок.

---

## Changes

- `apps/accounts/admin.py` — добавлен кастомный `UserAdmin` c `approve_selected_users`, `list_display`, `list_filter`, `ordering`, `search_fields`.
- `apps/accounts/signals.py` — добавлены `pre_save`/`post_save` receivers для отслеживания перехода `is_active: False → True`, восстановления `Profile` при необходимости и добавления в группу `player`.
- `apps/accounts/tests/test_admin.py` — добавлены тесты регистрации кастомного admin, структуры списка и approve action.
- `apps/accounts/tests/test_signals.py` — добавлены тесты активационного сигнала, восстановления отсутствующего профиля и отсутствия ложных срабатываний на обычном сохранении активного пользователя.

## Decisions made

- Логику добавления в группу `player` оставил в сигнале активации, а не переносил в admin action, потому что ADR-0004 прямо фиксирует это как следствие перехода `False → True`, а не как деталь только админского сценария.
- Создание `Profile` при активации сделано через `get_or_create(...)`, чтобы approve оставался устойчивым даже если профиль был удалён вручную или не был создан по внешней причине.
- Для детекции перехода `False → True` использован `pre_save` + `post_save`, чтобы сигнал не срабатывал на каждое сохранение уже активного пользователя.
