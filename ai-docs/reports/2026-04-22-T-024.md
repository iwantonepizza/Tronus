## Summary

Для `games` создан слой selectors: список сессий с фильтрами, detail selector с предзагрузкой участников и outcome, выборка будущих planned-сессий и recent completed. `get_session_detail` зафиксирован тестом на query-budget: полный detail кейс с 8 участниками укладывается в 3 SQL-запроса, то есть лучше требуемого лимита `<= 4`.

## Decisions made

- Для detail selector использован `Prefetch("participations", queryset=Participation.objects.select_related("user__profile", "faction"))` плюс отдельный prefetch `outcome`. Это даёт стабильный query-budget без N+1 на `profile` и `faction`.
- Базовый queryset сессий сделан с `select_related("mode", "deck", "created_by")` и стандартным ordering по `-scheduled_at, -pk`, чтобы его можно было напрямую переиспользовать в следующем `T-025`.
- `list_planned_after` возвращает future planned-сессии в естественном chronological order (`scheduled_at`, `pk`), а не в reverse order, потому что этот selector предназначен именно для upcoming view/use cases.

## Deviations from task

- Acceptance требовал `<= 4` SQL-запросов для `get_session_detail`; фактический результат в тесте — `3` запроса. Специально не подгонял до ровно 4, потому что меньшее число запросов здесь строго лучше и не меняет контракт selector’а.
