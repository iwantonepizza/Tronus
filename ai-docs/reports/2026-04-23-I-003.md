## Summary

Added a separate draft production-like deployment contour under `deploy/` without touching the working dev stack in `backend/`. The draft provides a same-origin nginx entrypoint, frontend static build image, backend container, Postgres, and example env files for local production-style smoke.

---

## Changes

- `deploy/docker-compose.prod.yml` — draft stack for `frontend + backend + db` with named volumes for `staticfiles` and `media`.
- `deploy/nginx/tronus.conf` — nginx reverse-proxy for `/api/` and `/admin/`, direct serving for `/static/` and `/media/`, SPA fallback for frontend routes.
- `deploy/env/backend.prod.env.example` — example backend/prod environment.
- `deploy/env/frontend.prod.env.example` — example frontend build environment.
- `deploy/README.md` — quick usage notes and caveats.
- `frontend/Dockerfile.prod` — multi-stage `Vite build -> nginx` production image.

## Tests

---

## Decisions made

- Kept deploy config separate from `backend/docker-compose.yml`, because that file is the active dev workflow and should not absorb production assumptions.
- Used variant A from `INTEGRATION_PLAN.md`: same-origin nginx in front of both frontend and backend, because it matches the existing CSRF/session model and avoids introducing new architecture.
- Left backend runtime on Django `runserver` inside the draft stack. This is intentionally temporary because adding `gunicorn` or `uvicorn` would require a dependency decision outside the current scope.
