## Decisions made

- Основной dev-сценарий оставлен cross-origin, а не через Vite proxy, потому что это прямо следует из `CR-005` и лучше приближает local flow к prod split-origin варианту.
- `/auth/csrf/` сделан в `accounts.views`, а не в отдельном integration app, потому что endpoint напрямую относится к auth flow и уже описан в `INTEGRATION_PLAN.md` внутри раздела auth.

---

## Deviations from task

- Runtime acceptance выполнен не через Docker/Postgres, а через временную SQLite fallback-БД и локальный `runserver`, потому что Docker daemon на машине выключен, а локальный Postgres не принимает текущие креды. Это не мешает проверить именно CORS/CSRF-поведение HTTP-слоя.
- `ai-docs/API_CONTRACT.md` пришлось обновить через аккуратный shell replace, а не через `apply_patch`, потому что файл в текущем состоянии плохо матчился патчером из-за кодировки/текстового состояния. Изменение точечное: добавлена одна строка про `/auth/csrf/`.

## How to verify

1. Перейти в `backend/`.
2. `python manage.py check`
3. `python manage.py makemigrations --check --dry-run`
4. `python -m pytest apps/accounts/tests/test_api.py -q -p no:asyncio`
5. `python -m pytest -q -p no:asyncio`
6. `docker compose config`
7. Поднять backend на рабочей БД или SQLite fallback:
   `DATABASE_URL=sqlite:///... python manage.py migrate`
   `DATABASE_URL=sqlite:///... python manage.py runserver 127.0.0.1:8000 --noreload`
8. Выполнить:
   `curl -i -H "Origin: http://localhost:5173" http://127.0.0.1:8000/api/v1/auth/csrf/`
9. Выполнить:
   `curl -i -X OPTIONS -H "Origin: http://localhost:5173" -H "Access-Control-Request-Method: POST" -H "Access-Control-Request-Headers: content-type,x-csrftoken" http://127.0.0.1:8000/api/v1/auth/login/`
10. Выполнить:
   `curl -i -H "Origin: http://localhost:5173" http://127.0.0.1:8000/api/v1/reference/factions/`
