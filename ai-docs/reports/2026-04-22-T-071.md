# Report for T-071: harden accounts register/login

**Agent:** codex
**Date:** 2026-04-22
**Status:** `completed`

---

## Summary

`accounts.register_user` больше не зависит от `post_save(User)` для создания `Profile`: сервис теперь создаёт или обновляет профиль явно. `LoginView` упрощён до жёсткого порядка проверок `user exists -> password valid -> is_active -> authenticate`, а regression покрыт отдельными тестами на отключённый сигнал и неизвестный email.

---

## Changes

- `backend/apps/accounts/services.py` — `register_user` переведён с молчаливого `Profile.objects.filter(...).update(...)` на явный `Profile.objects.update_or_create(...)`.
- `backend/apps/accounts/views.py` — `LoginView.post` больше не вызывает `authenticate` для несуществующего email и отдельно обрабатывает неверный пароль до проверки `is_active`.
- `backend/apps/accounts/tests/test_services.py` — добавлен regression test `test_register_user_works_without_signals`.
- `backend/apps/accounts/tests/test_api.py` — добавлен regression test, что login для неизвестного email возвращает `invalid_credentials` и не вызывает `authenticate`.

## Decisions made

- Сигнал `create_profile_for_user` оставлен как fallback для создания `User` вне `register_user` и не переносился в сервис полностью, потому что это прямо соответствует CR-003 и уже покрывает admin / management сценарии.
- В login flow сохранён финальный вызов `authenticate(...)` даже после `check_password()`, чтобы не обходить стандартный Django login backend и не вшивать логику аутентификации во view.
