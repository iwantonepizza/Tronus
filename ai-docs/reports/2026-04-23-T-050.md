## Summary

Implemented the full `avatars` MVP in `backend/`: model, migrations, Pillow-based generation, private API, and synchronization with `Profile.current_avatar`. The backend now stores original uploads and generated PNG avatars under `MEDIA_ROOT`, can mark one avatar as current, and can delete avatar history cleanly.

## Decisions made

- Stored avatar history in a separate `AvatarAsset` model and kept `Profile.current_avatar` as the single selected pointer, because this matches the data model and keeps public profile serialization cheap.
- Kept thumbnails as sidecar files (`*_thumb.png`) without a dedicated DB field, because ADR-0007 requires thumbnail generation but current API/docs do not expose a thumbnail field yet.
- Enforced one current avatar per user with a conditional DB constraint on `AvatarAsset.is_current`, then synchronized that flag from services to avoid drifting state.
