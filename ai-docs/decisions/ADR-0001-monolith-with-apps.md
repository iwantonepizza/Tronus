# ADR-0001: Django monolith с apps-based разбиением

**Status:** accepted
**Date:** 2026-04-21
**Deciders:** architect, owner

---

## Context

Проект — учёт партий настольной игры для компании друзей. Нагрузка низкая (десятки партий в месяц, десятки пользователей). Команда — один owner + AI-агенты для написания кода.

Нужно выбрать архитектурный подход: монолит / модульный монолит / микросервисы.

---

## Decision

Используем **Django monolith с разбиением на apps**. Каждая аппа имеет чёткую зону ответственности и публичный интерфейс (`services` / `selectors`). Импорт чужих `models` допустим только как FK-references.

---

## Alternatives considered

| Вариант | Плюсы | Минусы | Почему не выбран |
|---------|-------|--------|------------------|
| Микросервисы | Независимое развёртывание | Огромный overhead для команды из 1 человека, distributed tracing, coordination | Избыточно для масштаба. |
| Один большой app | Минимум структуры | Перемешивание ответственностей, невозможно ревьюить изменения | Проверено болью в Django-проектах. |
| FastAPI + SQLAlchemy | Легче API-first | Нет admin, нет миграций из коробки, руками писать всё | Django admin покрывает кейсы модерации и ручной правки, это критично. |

---

## Consequences

### Positive
- Быстрый старт, минимум инфраструктуры.
- Admin бесплатно.
- Migrations, ORM, auth из коробки.
- Apps-границы дают понятный unit ревью.

### Negative / Trade-offs
- Дисциплина импортов не enforced технически — нужны конвенции и ревью.
- Shared DB — отсутствие физической изоляции между модулями.

### Neutral
- Модель может эволюционировать в модульный монолит с более жёсткими границами, если понадобится.

---

## Revisit when

- Команда вырастает до 3+ разработчиков с параллельными потоками.
- Нагрузка требует горизонтального масштабирования конкретной аппы.
- Появляется требование independent deployment для какой-то подсистемы.
