# ADR-0002: Сервисный слой, тонкие views, без fat models

**Status:** accepted
**Date:** 2026-04-21
**Deciders:** architect

---

## Context

В Django есть два доминирующих подхода:

1. **Fat models** — бизнес-логика на моделях как методы.
2. **Service layer** — модели хранят только данные, логика в отдельных модулях `services.py` / `selectors.py`.

Проект будет расти фичами (голоса с ограничениями, финализация с инвариантами, статистика с агрегациями). Логику надо держать отдельно от моделей и views.

---

## Decision

**Service layer** подход (HackSoft-style):

- `models.py` — только данные + db-level constraints.
- `services.py` — мутации, транзакции, инварианты.
- `selectors.py` — чтение, агрегации, без сайд-эффектов.
- `views.py` — тонкие, делегируют в services/selectors.
- `serializers.py` — только shape данных для DRF.

---

## Alternatives considered

| Вариант | Плюсы | Минусы | Почему не выбран |
|---------|-------|--------|------------------|
| Fat models | Близко к «Django way» из туториалов | Невозможно тестировать логику отдельно от ORM, модели разрастаются до неюзабельных | Болезненный опыт на прошлых проектах. |
| Fat views | Быстро писать прототипы | Бизнес-логика привязана к HTTP, не переиспользуется, сложно тестировать | Антипаттерн. |
| Repository / DDD с entities | Полная изоляция | Overkill для такого масштаба, ORM всё равно торчит | Нет задач, оправдывающих сложность. |

---

## Consequences

### Positive
- Логика легко тестируется без HTTP.
- Views остаются маленькими — ревью проще.
- Агенту даётся ясная инструкция куда писать код.
- Переиспользование: один сервис дёргается и из API, и из management-команды, и из admin-action.

### Negative / Trade-offs
- Больше файлов, чем в «наивном» Django.
- Новичку нужно объяснять, почему `model.save()` вызывается из сервиса, а не из view.

### Neutral
- Соответствует известному стандарту (HackSoft Styleguide), легко онбордить людей.

---

## Revisit when

- Появится отдельная команда фронта, требующая BFF-паттернов — сервис-слой это переживёт.
- Переход на CQRS или event sourcing — тогда selectors/services логически разделены уже сейчас, переход проще.
