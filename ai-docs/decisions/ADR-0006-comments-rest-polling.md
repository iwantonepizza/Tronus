# ADR-0006: Комментарии под матчем через REST + polling

**Status:** accepted
**Date:** 2026-04-21
**Deciders:** architect, owner

---

## Context

Под каждым матчем есть «чат» для обсуждения. Возможные подходы: WebSocket (Django Channels), long-polling, обычный REST с periodic refetch.

---

## Decision

**Обычный REST.** Фронт делает `GET /api/v1/sessions/<id>/comments/` при открытии страницы и рефетчит:

- При возврате фокуса на вкладку (`visibilitychange`).
- После того как пользователь сам отправил комментарий.
- Опционально: interval-polling раз в 30–60 секунд, если страница в фокусе.

Бэк **ничего не знает о polling** — просто отдаёт список. Это чисто клиентская стратегия.

Никакого WebSocket, никакого Django Channels, никакого `asgi` кроме дефолтного.

---

## Alternatives considered

| Вариант | Плюсы | Минусы | Почему не выбран |
|---------|-------|--------|------------------|
| Django Channels + WebSocket | Честный realtime | Требует ASGI-сервер (Daphne / Uvicorn), Redis как channel layer, двойной deploy — web + worker. Для closed-group с максимум 5 одновременных читателей это несоразмерно | Overkill. |
| Server-Sent Events | Проще, чем WS, однонаправленные | Django из коробки не умеет, нужны костыли | Не выигрывает против polling. |
| Long-polling | Экономия запросов vs interval-polling | Держит соединение открытым, проблема с таймаутами балансировщика, клиент-код сложнее | Для 5 человек экономия копеечная. |
| Клиент сам решает (наш выбор) | Простейшая реализация, стандартный Django sync | Сообщения появляются с задержкой до 60с | Задержка приемлема для закрытой офлайн-игры. |

---

## Consequences

### Positive
- Phase 1 остаётся на sync Django без ASGI.
- Нет дополнительной инфраструктуры (Redis не нужен для этого).
- Фронту легко реализовать — стандартный `useEffect` + refetch.

### Negative / Trade-offs
- Задержка до ~60с. Для игры постфактум — не проблема.
- Interval-polling от нескольких клиентов = N запросов в минуту, где N = количество одновременно открытых страниц матча. В закрытой компании это ≤5. Приемлемо.

### Neutral
- Если когда-то понадобится realtime (например, live-коэффициенты или онлайн-трансляция партии) — добавляем Channels в отдельной аппе, не переделываем `comments`.

---

## Implementation notes

- API уже описан в `API_CONTRACT.md`.
- `GET /sessions/<id>/comments/?since=<comment_id>` можно добавить как оптимизацию в Phase 2 — отдавать только новые, если клиент уже имеет часть. Пока делаем полный список с пагинацией.
- Pagination: cursor-based по `created_at desc`, limit 50. Бесконечный скролл на фронте покажет старые.

---

## Revisit when

- Появится запрос на «видеть как друг пишет в реальном времени» — добавляем WebSocket.
- Количество активных читателей партии вырастет до десятков — polling станет шумным.
