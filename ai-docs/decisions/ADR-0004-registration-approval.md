# ADR-0004: Регистрация по email с approve от owner

**Status:** accepted
**Date:** 2026-04-21
**Deciders:** architect, owner

---

## Context

Сервис для закрытой компании, но owner не хочет раздавать invite-токены руками. Нужен поток, где:

1. Новый человек регистрируется сам (email + пароль + ник).
2. Owner видит «pending» заявку и апрувит её.
3. До апрува пользователь не может войти.

---

## Decision

Используем **стандартный Django-флоу через `is_active`**:

- `POST /auth/register/` создаёт `User(is_active=False)`.
- Django login автоматически отказывает `is_active=False` пользователям — защита встроенная, без дополнительного кода.
- В Django admin owner видит список `is_active=False` и нажимает кастомный action `Approve selected users` — сервис `accounts.services.approve_user(user)` ставит `is_active=True`.
- Email-верификация **не делается** в MVP. Owner и так знает email друга; если зарегистрировался не тот — owner просто не апрувит.
- После апрува — сигнал (Django signal) на `User.is_active` переход `False → True` создаёт `Profile` (если ещё нет) и группу `player`.

---

## Alternatives considered

| Вариант | Плюсы | Минусы | Почему не выбран |
|---------|-------|--------|------------------|
| Invite-token (прошлый план) | Нет очереди на апрув | Owner должен генерить и раздавать токены вручную | Owner сам отказался. |
| Регистрация с email-верификацией + approve | Канонично | Нужна email-инфраструктура в MVP | Overkill для closed-group. |
| Отдельное поле `is_approved` помимо `is_active` | Чистая семантика | Два флага вместо одного, путаница | Django уже имеет `is_active` с нужной семантикой. |
| Owner создаёт юзеров вручную в админке | Максимум контроля | Трение: при каждом друге owner лезет в админку, придумывает пароль, передаёт | Против первого ответа owner'а. |

---

## Consequences

### Positive
- Нулевая инфраструктура (никакой email-сервер, никакие токены).
- Django-нативный подход, `login_required` / auth-bэкенды работают без кастомизации.
- Легко мониторить через admin filter `is_active=False`.

### Negative / Trade-offs
- Нет автоматической защиты от typosquatting email (зарегистрировался как `frend@` вместо `friend@`) — ловим на этапе апрува.
- Если `is_active=False` используется где-то ещё для других целей (например, soft-ban) — потребуется второй флаг. Пока таких кейсов нет.

### Neutral
- При отказе в апруве просто удаляем user (или оставляем `is_active=False`, никогда не апрувим).

---

## Revisit when

- Регистраций станет столько, что ручной апрув станет затыком (маловероятно для closed-group).
- Появится требование email-верификации (например, для восстановления пароля через email) — тогда добавляем флоу, но `is_active` остаётся master-ключом доступа.
